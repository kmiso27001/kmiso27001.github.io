實戰篇：不設防的 Cookie

🍪 為什麼 Cookie 需要 HttpOnly？

🎭 身份憑證暴露

預設情況下，瀏覽器允許 JavaScript 透過 document.cookie 讀取 Cookie。若其中包含 Session ID，一旦被竊取，駭客即可冒充用戶身份。

🕷️ XSS 聯集攻擊

如果網站存在 XSS (跨站腳本) 漏洞，駭客可以注入惡意腳本讀取使用者的 Cookie，並將其傳送至自己的伺服器，造成會話劫持 (Session Hijacking)。

🛡️ 關鍵的一道鎖

HttpOnly 標籤能指示瀏覽器「禁止前端 JavaScript 存取該 Cookie」。這雖然不能阻止 XSS 的發生，但能阻斷駭客最致命的目標：竊取會話憑證。

⚖️ 程式碼大對決：HttpOnly 標籤設定

PHP (危險的配置方式 ❌) 

// 🚨 錯誤 1：直接使用預設 session_start()，且 php.ini 未配置安全選項
session_start();

// 🚨 錯誤 2：手動設置 Cookie 時忽略了 HttpOnly 參數
// 參數清單中最後兩個 boolean 代表 Secure 與 HttpOnly，此處皆為預設的 false
setcookie("user_auth", "token_value", time()+3600, "/");
PHP (安全的配置方式 ✅) 

// ✨ 正確 A：全局配置 (推薦)，在 session_start() 之前設置
session_start([
    'cookie_httponly' => true,
    'cookie_secure' => true, // 僅透過 HTTPS 傳輸
    'samesite' => 'Lax'      // 防範 CSRF
]);

// ✨ 正確 B：手動設置特定 Cookie
$name = "user_preference";
$value = "dark_mode";
$expire = time() + 3600;
$path = "/";
$domain = "";
$secure = true;   // 僅 HTTPS
$httponly = true; // 關鍵：禁止 JavaScript 存取
setcookie($name, $value, $expire, $path, $domain, $secure, $httponly);

🛡️ 開發建議：全方位保護 Session

🧱

深度防禦 (Defense in Depth)

HttpOnly 並不能防止 XSS 攻擊本身，但能大幅降低 XSS 發生後的損害（防止 Session 被盜）。應與 CSP 策略、輸入過濾並行使用。

🔒

強制 Secure 屬性

除了 HttpOnly，對於身份憑證應同時開啟 Secure 標籤，確保 Cookie 不會在不安全的 HTTP 環境下以明文傳輸，防止中間人攻擊。

⏱️

Session 過期與重建機制

縮短 Session 的有效時間 (TTL)，並在用戶登入/登出、提權時重新生成 Session ID (session_regenerate_id())，降低憑證被利用的窗口期。

🔍

定期審計與檢查

使用瀏覽器開發者工具的 "Application" 面板或 OWASP ZAP 掃描，定期檢查所有敏感 Cookie 的 HttpOnly 與 Secure 欄位是否已被正確設定。
