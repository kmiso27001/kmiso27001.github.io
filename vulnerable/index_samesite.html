實戰篇：欠缺 SameSite 屬性

🏷️ 什麼是 SameSite 屬性？

🎭 隱性的跨站信任

若未設定 SameSite，舊版瀏覽器預設會在所有跨站請求 (Cross-Site Requests) 中附帶 Cookie。這表示駭客能利用您的登入狀態在別的網站發起惡意請求 (CSRF)。

🛡️ SameSite 的三種模式

• Strict：最嚴格，僅限同站請求攜帶。
• Lax：允許安全頂級導航攜帶，阻擋跨站 POST。
• None：完全允許跨站，但必須搭配 Secure 屬性。

⚡ 從底層阻斷 CSRF

相較於傳統的 Anti-CSRF Token，配置 SameSite 屬性可以從瀏覽器底層直接阻斷非預期的跨站請求，是現代 Web 應用最基礎的防禦防線。

⚖️ 程式碼大對決：SameSite 屬性設定

PHP (危險的配置方式 ❌) 

// 🚨 危險的手動設置方式：僅定義了名稱、值、過期時間與路徑
setcookie("session_id", "xyz789", time()+3600, "/");

// 伺服器回傳的 Header 將呈現：
// Set-Cookie: session_id=xyz789; expires=...; path=/
// 缺少 SameSite 屬性，舊版瀏覽器將暴露於 CSRF 風險下
PHP 7.3+ (安全的配置方式 ✅) 

// ✨ 現代 PHP 支援以陣列方式精確設定 Cookie 屬性
$cookie_options = [
    'expires' => time() + 3600,
    'path' => '/',
    'domain' => '',
    'secure' => true,
    'httponly' => true,
    'samesite' => 'Strict' // 關鍵：防禦跨站請求偽造
];
setcookie("auth_token", "secure_value", $cookie_options);

// 若為全局 Session 配置：
// session_start(['cookie_samesite' => 'Lax', ...]);

🛡️ 開發建議：SameSite 配置實務

👍

採納「預設 Lax」原則

雖然 Chrome 80+ 已預設將未標註的 Cookie 視為 Lax，但為了確保舊版瀏覽器的一致性，開發者應明確在代碼中宣告 SameSite 屬性。

⚖️

區分業務需求 (Strict vs Lax)

一般展示性頁面使用 Lax，確保用戶從外部點擊連結進入時仍保持登入；但對於金流、密碼變更等高風險模組，應強制使用 Strict 阻斷所有跨站請求。

🧩

處理跨站第三方資源 (None)

若功能 (如金流 iframe、SSO 單一登入) 確實需要跨站傳遞 Cookie，必須設為 `SameSite=None`，並且強制開啟 `Secure` 屬性 (僅限 HTTPS)。

🔄

舊系統漸進式升級

針對無法升級 PHP 的舊系統，可考慮在 Web 伺服器層級 (如 Nginx 的 proxy_cookie_path 或 Apache 的 Header 規則) 統一注入 SameSite 屬性。
