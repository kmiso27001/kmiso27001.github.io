實戰篇：MIME 嗅探危機

🐶 什麼是 MIME 嗅探 (MIME Sniffing)？

🎭 偽裝的檔案

駭客上傳了一個名為 avatar.jpg 或 user_bio.txt 的檔案，但檔案內容實際上是惡意的 JavaScript 代碼。

🕵️ 瀏覽器的自動猜測

當這個檔案被其他網頁以 <script src="..."> 引用時，如果伺服器沒有強制規定 MIME 類型，瀏覽器會自作主張去「嗅探」檔案內容，發現是程式碼後就會執行它。

🛑 nosniff 的防禦力

加入 X-Content-Type-Options: nosniff 標頭可以強制瀏覽器「絕對遵守」伺服器宣告的類型。若類型是 text/plain，瀏覽器就會拒絕把它當作腳本執行，封殺此類 XSS 攻擊。

⚖️ 程式碼與系統對決：如何禁用 MIME 嗅探？

PHP (危險的配置方式 ❌) 

// 🚨 危險：僅設置了 Content-Type，未限制瀏覽器的嗅探行為
// 即使宣告為 text/plain，若內容包含 JS，瀏覽器仍可能執行它
header("Content-Type: text/plain");
echo $user_uploaded_file_content;
伺服器設定 (安全的防護方式 ✅) 

// ✨ PHP 安全寫法：強制加上 nosniff 標記
header("X-Content-Type-Options: nosniff");
header("Content-Type: text/plain; charset=UTF-8");
echo $user_uploaded_file_content;

// ✨ 或在 Web 伺服器 (Apache) 全局配置：
// Header set X-Content-Type-Options "nosniff"

// ✨ Nginx 全局配置：
// add_header X-Content-Type-Options "nosniff" always;

🛡️ 開發與維運建議：資源類型的嚴格管控

🗂️

搭配正確的 MIME 類型

設置 nosniff 的前提是伺服器必須精確回傳正確的 Content-Type。例如，CSS 檔案必須宣告為 `text/css`，否則在 nosniff 開啟的情況下，瀏覽器會拒絕加載樣式。

🔍

上傳檔案的深度過濾

針對上傳功能，應檢查檔案「真實內容」是否與副檔名相符。對圖片檔案可進行重新縮放或轉檔，移除內嵌的惡意元數據 (Metadata) 或腳本代碼。

🧱

與 CSP 縱深防禦結合

此標頭應與 Content Security Policy (CSP) 配合使用。即便駭客繞過了 MIME 限制，CSP 也能從根本上限制惡意腳本的執行來源。

✅

納入資安基準配置

缺少 nosniff 通常被掃描工具視為中低風險，但在資管稽核流程中，應將其列為網頁伺服器基準配置 (Baseline Configuration) 的全站必備標頭。
