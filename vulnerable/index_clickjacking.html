實戰篇：Missing Anti-clickjacking Header

🖱️ 什麼是點擊劫持 (Clickjacking)？

🖼️ 隱形的陷阱

駭客將受信任的目標網站（例如您的網路銀行或社群帳號設定頁）透過 <iframe> 嵌入到惡意網站中，並將其設為完全透明。

🎁 偷樑換柱

駭客在透明的目標網站底下，放了一個極具誘惑力的按鈕（例如「點擊領獎」）。因為目標網站是透明的，受害者以為自己點的是領獎按鈕。

💥 誤觸核彈

由於透明的目標網站在最上層，滑鼠點擊實際上穿透了誘餌，點到了隱藏的「刪除帳號」或「轉帳」按鈕！這一切都是在您的登入狀態下發生的。

🕹️ 實戰演練：親自體驗被劫持的感覺

下方模擬了一個駭客建立的抽獎網站，該網站巧妙地將「銀行網站的刪除帳號頁面」隱藏並覆蓋在「領取獎品」按鈕的上方。請試著先直接點擊看看，然後再勾選「揭穿魔法」觀察駭客是怎麼做到的！

❌ 測試危險配置 (無安全標頭防護)

🎉 恭喜您成為第 100 萬位訪客！ 🎉

▲ 因為目標網站缺乏防護，被成功嵌套。點擊將誤觸「刪除帳號」！

✅ 測試安全配置 (有 X-Frame-Options)

🎉 恭喜您成為第 100 萬位訪客！ 🎉

▲ 目標網站使用了安全標頭，瀏覽器直接拒絕載入 iframe，攻擊失效！

⚖️ 程式碼大對決：Clickjacking 防禦

clickjacking_vulnerable.php (無防護 ❌) 

<?php
session_start();
// 🚨 危險：沒有設定任何反框架化的 HTTP 標頭

if (!isset($_SESSION['user_id'])) {
    die("請先登入");
}
?>

<!-- 駭客可以任意在惡意網站使用 iframe 嵌入此頁面 -->
<iframe src="vulnerable.php" style="opacity: 0"></iframe>
clickjacking_secure.php (安全標頭 ✅) 

<?php
session_start();

// ✨ 安全做法 A：使用傳統的 X-Frame-Options 標頭
header("X-Frame-Options: DENY");

// ✨ 安全做法 B：使用現代的 Content Security Policy 標頭
header("Content-Security-Policy: frame-ancestors 'none';");
?>

<!-- 瀏覽器會嚴格遵守標頭指示，拒絕在 iframe 中載入此頁面 -->

🛡️ 開發建議：防範點擊劫持

🚫

優先採用 DENY 策略

對於包含敏感操作（轉帳、刪除、修改密碼）的頁面，應強制設定 X-Frame-Options 為 DENY，完全禁止被嵌入。

⚙️

統一標頭管理

建議在 Web 伺服器層級（如 Apache 的 .htaccess 或 Nginx.conf）全域配置此標頭，確保開發者不會因為遺漏個別 PHP 檔案而產生漏洞。

🗑️

淘汰舊式 JS 腳本

過去常使用 JavaScript (如 if (top != self) ...) 來檢測嵌套，但這容易被駭客繞過，應改為依賴標準的 HTTP 安全標頭。

🛡️

縱深防禦機制

點擊劫持常與 CSRF 結合使用，因此應同時實施 Anti-CSRF Tokens！即使受害者被騙點擊，也會因為請求中沒有正確的 Token 而導致失敗。
