實戰篇：伺服器資訊洩漏

📡 什麼是 X-Powered-By 資訊洩漏？

🕵️ 駭客的偵察階段

在發動真實攻擊前，駭客會進行「偵察 (Reconnaissance)」，試圖摸清目標網站使用的是什麼技術、框架與伺服器軟體，以便精準挑選攻擊武器。

🏷️ 過度誠實的標頭

X-Powered-By 與 Server 是 HTTP 回應標頭。預設情況下，許多後端語言 (如 PHP) 會自動在標頭中標示自己精確的版本號 (如 PHP/5.6.40)。

🎯 致命的弱點比對

一旦獲得精確的版本號碼，攻擊者可以輕易地在 CVE 漏洞資料庫中比對該版本是否含有未修補的遠端代碼執行 (RCE) 弱點，直接一擊必殺。

🕹️ 實戰演練：自動化掃描器模擬

下方模擬了駭客使用自動化掃描工具 (如 curl 或 Nmap) 對伺服器發送 HTTP 請求。請點擊「😈 啟動掃描器」來觀看伺服器回傳的 HTTP Headers，看看會洩漏什麼資訊！

❌ 測試危險配置 (預設暴露版本)
> curl -I https://shop.com/api
等待掃描...
✅ 測試安全配置 (隱藏標頭)
> curl -I https://shop.com/api
等待掃描...

⚖️ 程式碼與系統對決：如何隱藏標頭？

php.ini (危險的預設配置 ❌) 

; PHP 預設情況下會開啟 expose_php
; 這會自動在所有 HTTP 回應中加入 X-Powered-By 標頭
expose_php = On

; Web 伺服器 (Apache/Nginx) 預設也會輸出詳細版本
; ServerTokens OS  (在 Apache 中)
系統配置 (安全的防護方式 ✅) 

; ✨ 最根本的做法：在 php.ini 中將其關閉
expose_php = Off

; 另外，若無法修改 php.ini，也可在 PHP 代碼中移除：
; header_remove('X-Powered-By');

; ✨ 同步隱藏 Apache 伺服器資訊 (於 httpd.conf)：
ServerTokens Prod
ServerSignature Off

🛡️ 開發與維運建議：隱晦式安全 (Security through obscurity)

🥷

落實模糊化原則

雖然隱藏資訊並不能修復漏洞，但能顯著提升駭客的攻擊成本與時間，減少被大規模自動化掃描器 (Bots) 盲目鎖定與攻擊的機率。

🚫

全面移除冗餘標頭

除了 X-Powered-By 與 Server 標頭外，也應注意 `X-AspNet-Version`、框架特有標頭 (如 `X-Generator`) 等無助於功能但會洩漏技術棧的資訊。

🔄

保持系統更新 (治本之道)

資訊隱藏是「防君子不防小人」的第一道防線，核心防禦依然是保持作業系統、Web 伺服器與 PHP 處於官方維護支援的生命週期內，並定時安裝安全性更新。

🔍

利用 ZAP 或 Nessus 稽核

將「標頭資訊洩漏」納入 CI/CD 的自動化資安掃描流程中，確保每次伺服器環境升級或搬遷時，這些敏感標頭不會悄悄地被重新開啟。
