實戰篇：Parameter Tampering

🕵️ 什麼是參數篡改 (Parameter Tampering)？

🎭 過度信任客戶端

開發者有時會把重要數據（如商品價格、權限等級）放在隱藏表單 <input type="hidden"> 或是 URL 參數中，以為使用者看不到就不會改。

🛠️ F12 開發者工具

駭客根本不需要寫高深的程式，只要按下瀏覽器的 F12 鍵打開開發者工具，就能輕易把隱藏的價格改成 1 元，再按下送出。

💥 後端未經二次校驗

如果伺服器後端收到訂單後，直接使用前端傳來的價格進行結帳，而沒有去資料庫重新查詢真實價格，就會產生嚴重的財務損失！

⚖️ 程式碼大對決：參數驗證

param_vulnerable.php (信任前端 ❌) 

$item_id = $_POST['item_id'];

// 🚨 致命錯誤：直接信任前端傳來的價格！
$price = $_POST['price']; 
$qty = $_POST['qty'];

$total = $price * $qty;
// 駭客把 $_POST['price'] 改成 1，公司就虧大了
process_payment($_SESSION['user_id'], $total);
param_secure.php (後端為準 ✅) 

$item_id = (int)$_POST['item_id'];
$qty = (int)$_POST['qty'];

// ✨ 安全做法：只拿 ID，其餘敏感數據一律向資料庫查詢
$stmt = $pdo->prepare("SELECT price FROM products WHERE id = ?");
$stmt->execute([$item_id]);
$product = $stmt->fetch();

$real_price = $product['price'];
$total = $real_price * $qty;
process_payment($_SESSION['user_id'], $total);

🛡️ 開發建議：防範參數篡改

👑

後端權威原則

涉及金流、權限或業務邏輯的數據，絕對必須在伺服器端進行二次校驗。前端傳來的參數只能視為「意圖」，不能當作「事實」。

🔍

輸入驗證與過濾

嚴格檢查所有參數的資料類型（是否為數字）、範圍（數量不可為負數）。防堵各種邊界測試與異常輸入。

📦

善用 Session

敏感的狀態資料（如購物車總額、用戶等級）應存放在伺服器端的 $_SESSION 中，而不是寫在 Cookie 或是隱藏表單裡。

🔑

數位簽章保護

若某些情境非得在前端傳遞狀態（例如付款回調的參數），應使用 HMAC 雜湊演算法加上 Secret Key 來對參數進行數位簽章，防止被篡改。
