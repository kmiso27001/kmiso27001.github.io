實戰篇：脆弱的 JS 函式庫

🧩 什麼是 Vulnerable JS Library？

⏳ 年久失修的依賴

網頁常會引用 jQuery、Bootstrap 等開源套件，但開發者往往「引用一次就忘記了」，導致專案上線幾年後，使用的是充滿漏洞的老舊版本。

🎯 精確的已知弱點

駭客可以輕易查出你網站使用的 JS 版本（例如 jQuery 1.10.2），然後上 CVE 資料庫尋找該版本專屬的 XSS 攻擊 Payload 直接打擊。

💥 防線從內部崩潰

即使工程師自己寫的 PHP 或 JS 邏輯再安全，只要引用了脆弱的第三方套件，駭客就能利用這些套件底層的漏洞來執行惡意代碼。

🕹️ 實戰演練：老舊 jQuery 的 XSS 危機

下方模擬了一個網頁分別引用了「極老舊的 jQuery 1.10.2」與「最新安全的 jQuery 3.7.1」。請點擊「😈 發動惡意連結攻擊」來模擬駭客向這兩個頁面傳送一段夾帶惡意 Hash 的 URL，看看舊版 jQuery 是如何中招的！

✅ 測試安全配置 (jQuery 3.7.1)

⚖️ 套件引用大對決：版本與 SRI 驗證

HTML (危險的引用方式 ❌) 

<!-- 🚨 錯誤 1：使用極老舊且存在 CVE 漏洞的版本 -->
<!-- 🚨 錯誤 2：沒有 SRI (integrity) 驗證，若 CDN 被駭將直接淪陷 -->
<script src="https://code.jquery.com/jquery-1.10.2.min.js"></script>

<script>
    // 舊版 jQuery 處理 $() 時若傳入 HTML，會自動建構 DOM
    // 導致傳入 location.hash 會觸發 XSS
    $(location.hash);
</script>
HTML (安全的引用方式 ✅) 

<!-- ✨ 正確：使用官方長期支援的最新 LTS 版本 -->
<!-- ✨ 正確：加上 integrity 屬性，確保腳本內容未被竄改 -->
<script 
    src="https://code.jquery.com/jquery-3.7.1.min.js" 
    integrity="sha256-/JqT3SQfawRcv/BIHPThkBvs0OEvtFFmqPF/lYI/Cxo=" 
    crossorigin="anonymous">
</script>

<script>
    // 新版 jQuery 已修補此問題，無法再透過選擇器注入惡意標籤
    $(location.hash); 
</script>

🛡️ 開發建議：如何管理第三方套件？

📦

建立軟體材料清單 (SBOM)

記錄專案中所使用的所有第三方元件及其版本。這是資訊資產管理與合規性審核的基礎，當零日漏洞爆發時能快速盤點受害範圍。

🤖

啟用自動化弱點掃描

在 CI/CD 流程中整合自動化工具 (如 npm audit, GitHub Dependabot)，當依賴項出現漏洞時，系統會自動發出警報並建議修復版本。

🔒

使用子資源完整性 (SRI)

當從外部 CDN 引用資源時，務必包含 integrity 雜湊值。這能確保即使 CDN 伺服器遭到駭客入侵並竄改檔案，惡意代碼也會被瀏覽器拒絕執行。

🧱

實施 CSP 與 WAF 虛擬補丁

透過 CSP 嚴格限制腳本來源。若因系統架構過舊無法升級函式庫，可先在 Web 應用程式防火牆 (WAF) 上設定過濾規則攔截攻擊流量。
