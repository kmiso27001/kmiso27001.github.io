實戰篇：Absence of Anti-CSRF tokens

🎭 什麼是跨站請求偽造 (CSRF)？

👥 借刀殺人

CSRF 是一種迫使已驗證用戶在當前 Web 應用程式上執行非自願操作的攻擊手法。駭客利用您瀏覽器中已登入的 Session，借您的手發送請求。

🍪 核心問題：Cookie

瀏覽器發送請求時，會自動附帶與該域名相關的 Cookie（如 Session ID）。伺服器僅依賴 Cookie 驗證，無法區分請求是由用戶自主觸發，還是由第三方網站惡意偽造。

🎯 攻擊範例

攻擊者建構惡意網頁，內含隱藏表單：
<form action="bank.com/transfer" method="POST">
當受害者開啟惡意網頁，表單自動送出，瀏覽器帶上 Session Cookie，銀行誤以為是合法轉帳。

⚖️ 程式碼大對決：CSRF 防禦

csrf_transfer.php (未驗證來源 ❌) 

session_start();
// 🚨 危險：僅驗證 Session，未驗證請求來源的唯一性
if (isset($_SESSION['user_id'])) {
    $to = $_POST['to'];
    $amount = $_POST['amount'];
    
    // 執行轉帳邏輯（此處極易受 CSRF 攻擊）
    update_balance($_SESSION['user_id'], $to, $amount);
    echo "轉帳成功！";
}
secure_csrf_transfer.php (加入防護令牌 ✅) 

session_start();
// 步驟 A：生成並儲存 Token
if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}

if ($_SERVER['REQUEST_METHOD'] === 'POST') {
    // ✨ 安全：伺服器端驗證 Token
    if (!isset($_POST['csrf_token']) || 
        $_POST['csrf_token'] !== $_SESSION['csrf_token']) {
        die("CSRF 攻擊檢測：無效的請求來源。");
    }
    // 驗證通過，執行轉帳
}

🛡️ 防禦之道：多層次防禦策略

🔑

同步令牌 (Anti-CSRF Token)

確保每個狀態更改請求（POST/PUT/DELETE）都包含伺服器生成的隨機、不可預測 Token。

🍪

SameSite Cookie 屬性

將 Session Cookie 屬性設為 SameSite=Lax 或 Strict。限制瀏覽器在跨站請求時發送 Cookie。

🛠️

使用現代框架

如 Laravel、Django 等內建 CSRF 防護機制的框架，這些框架會自動處理 Token 的生成與校驗。

📱

雙重驗證 (2FA)

對於高風險操作（如修改密碼、大額轉帳），增加簡訊驗證碼或密碼二次確認，作為最後一道防線。
